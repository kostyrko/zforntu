Testowanie wprowadzenie

Opublikowano ptk 05 listopada 2021 w testing • 3 min read

Wstępne pojęcia

Dynamic testing - egzekucja kodu/stystemu/aplikacji

Static Testing - recenzowanie/sprawdzenie kodu (np. czy jest zgodny z standardami), sprawdzamy wobec wymagań (planu, dokumentacji)

Validation - czy produkt jest taki jakiego klient sobie zażyczył

Verification - czy produkt jest zbudowany zgodnie z zasadami (performance, security, design, etc.)

Test Process A - 1) Plan [Analysis, Strategy, Plan, Tools] 2) Design (Test cases, Scripts, Scenarios, Environments) 3) Execution (Test Reports, Defect Tracking, Defect Analysis, Reports)

Test process B - 1) Test planning 2) Test monitoring & control (compare progress with the planned, correction actions to follow the plan) 3) Test analysis (Finding conditions) 4) Test design (designing test cases i.e. dividing in to parts) 5) Test implementation (i.e. taking a look a an environment) 6) Test execution

Poziomy testów

Units - najbardziej podstawowe testy

Integration - integracja komponentów, interakcja pomiędzy komponentami -> dwa rodzaje testów: Component Integration/ Integracja pomiędzy komponentami, System Integration/ Integracja pomiędzy systemami (Web service + backend + frontend) [zadanie testera]

Systems - testowanie całego systemu [najwięcej zadań testerów odbywa się tutaj] - testowanie powinni odbywać się w takim środowisku w jakim będzie aplikacja stosowana. Szukanie jak najwięcej defektów.

Acceptance - sprawdzanie czy system działa w sposób poprawny - wersja Alpha oraz Beta [użytkownicy, lub klient]

Typy testów (wybrane)

Functional testing - testowanie funkcjonalności aplikacji - co robi i co powinien robić system (np. czy działa funkcjonalność logowania?)

Non-functional testing - testowanie wydajności aplikacji (czy czas odpowiedzi jest poprawny)

Black box testing - testowanie bez sprawdzania kodu

White box testing - sprawdzanie jak wygląda przepływ informacji wobec kodu poprzez aplikację

Dynamic testing - egzekucja kodu

Static testing - sprawdzanie wobec user stories // ogólny obraz aplikacji

Confirmation testing - sprawdzanie aplikacji po poprawkach

Regression testing - sprawdzanie aplikacji po zmianach, ale całości aplikacji aby wykryć ew. powiązania z częściami aplikacji które nie zostały pozornie zmienione

Smoke testing - scenariusz, który sprawdza główne części aplikacji a nie jej całość (szybki test ogólny, ogólnych i głównych części aplikacji)

Test case Writing

A set of preconditions, inputs, actions and expected results and postconditions developed on test conditions.

Example 1

Test condition - login page

Test case title: Verify login with a valid user name and password

Preconditions: User is already registered using valid credentials

Test steps: 1) enter a valid user name and password 2) click on login button

Expected result: User should be able redirected to admin page

Test Scenario (test suit): Login

Test environment: hardware and software

Actual result (fill in after executing a test case): User is redirected to admin page // same as expected

Status: New (ready to test), Pass (actual result is the same as expected), Fail, Blocked/Skipped (can't be executed)

Valid test case - a positive test case

Invalid test case - negatywny przykład testu, zachowanie się programu w przypadku nieporządnego zachowania się użytkownika np. 'sign-up without choosing type user' (where user type i required) -> expected result: a message should appear...albo za długa/krótka nazwa użytkownika, nazwa użytkownika już znajdującego się w bazie danych

Bug Reporting

1) Title 2) Steps to reproduce (all the data given) 3) Expected results 4) Actual results 5) Test environment 6) Screenshot or video (show the click) 7) Bug priority: Critical, High, Medium (portrait mode), Low (spelling)

Types of defects Functional, Visual (UI), Content, Performance, Suggestion, Security
